RAGBench: A Configurable RAG Evaluation Pipeline With an Application to Small Local Models
Noah Collin | March 2026 (Revised)

The Problem

There is no standard, reproducible way to answer the question: "which RAG configuration works best for my documents and queries?" A practitioner choosing between chunking strategies, embedding models, retrieval approaches, and language models faces a combinatorial space with no systematic tooling to explore it. Most published RAG comparisons hold three axes constant and vary one, making results hard to generalize. Meanwhile, users running models locally on consumer hardware face an additional constraint: they need configurations that are not just good, but good enough given limited compute.

This project addresses both problems. We build a configurable evaluation pipeline that runs the full cartesian product of RAG configurations against any corpus, scores the results, and trains a meta-learner to predict the optimal configuration for new queries. We then demonstrate the pipeline by running two controlled experiments on small local models: one testing whether RAG strategy sophistication can substitute for raw parameter count, and another testing whether chunking strategy — the way documents are split for retrieval — has a differential impact on small models compared to large ones.

What We're Building

The project has three components: a pluggable evaluation pipeline, a trained meta-learner, and a deployed recommendation endpoint.

The Evaluation Pipeline

The pipeline takes four pluggable axes as input — chunker, embedder, strategy, and language model — and runs every combination against a document corpus with generated evaluation queries. Each configuration produces an answer, which is scored by an automated judge. The result is a structured dataset of (configuration × query × document → scores) that can be analyzed, visualized, and used to train the meta-learner.

Each axis is defined by a Python Protocol (structural interface), so users can add their own implementations without modifying the framework:

Chunkers split documents into retrieval units. We include semantic chunking (respecting meaning boundaries), fixed-size chunking, recursive chunking, and sentence-level chunking. All share the same interface: text in, chunks out.

Embedders convert text to vector representations for retrieval. We include Ollama-hosted models (mxbai-embed-large), HuggingFace sentence-transformers (all-MiniLM-L6-v2), and Google's text-embedding-005 via API. A user could add any embedding backend by implementing three methods: name, dimension, and embed.

Strategies define the retrieval-and-reasoning pipeline. We implement five, described below.

Models are the language models that generate answers. Our primary family is Qwen3 via Ollama (0.6B, 1.7B, 4B, 8B), with Gemma 3 (1B, 4B) as cross-family validation. Gemma 3 overlaps with Qwen3 at 1B and 4B, enabling direct same-size comparisons across architectures to test whether findings generalize beyond a single model family.

Query generation is also pluggable. We use RAGAS (Shahul et al., 2023, arXiv:2309.15217) as the primary synthetic generator, with support for human-curated query sets as a validation anchor. Generated queries are classified by type — factoid, reasoning, multi-context, and conditional — and persisted as a static artifact for reproducibility. A round-trip consistency filter (inspired by Promptagator, Dai et al., 2022, arXiv:2209.11755) validates generated queries by checking that each query retrieves its source passage.

Scoring uses Claude as an LLM-as-judge, a methodology validated by Saad-Falcon et al. (2024, NAACL) in the ARES system. We score on faithfulness, relevance, and conciseness. Per the professor's feedback on LLM-as-judge noise, we plan to manually validate a sample of scores against human judgment before relying on them for training.

The Five Strategies

We test five RAG strategies. In Experiment 1 (below), chunker and embedder are held constant so that strategy and model size are the only variables.

Naive RAG — baseline. Retrieve chunks, generate answer. No special logic. Where raw model capability matters most.

Multi-Query RAG — generates multiple phrasings of the query, retrieves for each, merges results. Compensates for weak query understanding in small models by casting a wider retrieval net.

Corrective RAG (Shi et al., 2024, arXiv:2401.15884) — evaluates retrieved chunk relevance and discards low-confidence context before generation. Addresses small models' tendency to hallucinate from irrelevant context.

Adaptive RAG (Jeong et al., 2024, NAACL) — classifies query complexity and routes accordingly: skip retrieval for simple questions, single-pass for moderate, multi-pass for complex. Prevents unnecessary retrieval from confusing small models on easy queries.

Self-RAG (Asai et al., 2024, ICLR) — the model decides at each step whether retrieval is needed, evaluates what it retrieved, generates an answer, and critiques its own output. Implemented as a prompted approximation. Highest ceiling but requires self-reasoning capability that very small models may lack entirely.

Experiment 1: Strategy × Model Size

Our first research question: can RAG strategy sophistication substitute for raw parameter count?

We hold chunker (recursive, 512 tokens, 50-token overlap) and embedder (mxbai-embed-large) constant and run the 5 × 4 matrix of strategies × Qwen3 sizes, plus 5 × 2 for Gemma 3 at 1B and 4B. This gives us 30 configurations per query, scored across three quality dimensions, with enough data to identify interaction effects — cases where the ranking of strategies changes depending on model size — and cross-family validation at matched parameter counts.

The hypothesis is that strategies like Self-RAG and Corrective RAG disproportionately help small models by compensating for their weaknesses (poor self-monitoring, susceptibility to irrelevant context), while providing diminishing returns for larger models that handle these challenges natively. Finding the capability threshold where Self-RAG's self-critique becomes viable is one of the most interesting questions.

Experiment 2: Chunking Strategy × Model Size

Our second research question: does the optimal chunking strategy depend on model size?

Existing chunking comparisons (Wang et al., 2024; Chen et al., 2023) use large models (GPT-3.5/GPT-4) exclusively. No published work systematically studies how chunking strategy interacts with model size for small (<14B) models. There is indirect evidence that smaller models are more sensitive to chunking choices: Liu et al. (2023, "Lost in the Middle") demonstrated that small models degrade more severely with irrelevant context, suggesting that retrieval precision — which is directly affected by chunking — matters more for small models.

We hold strategy (Naive RAG, to isolate the chunking effect) and embedder constant, and run 3 chunking strategies × 4 Qwen3 sizes: fixed-size (512 tokens), recursive (512 tokens, 50-token overlap), and semantic chunking. This gives us 12 configurations per query — a small matrix that runs in a fraction of the time of Experiment 1.

The hypothesis is that smaller models benefit disproportionately from chunking strategies that maximize retrieval precision (semantic > recursive > fixed), while larger models are more robust to chunking differences because they handle irrelevant context better. If confirmed, this has practical implications: users running small local models should invest more effort in chunking than users running large models.

Features, Meta-Learner, and Deployment

From each (query, document, retriever) triple, we extract features before any model runs: query length, query type (one-hot), named entity count, document length, vocabulary entropy, mean and variance of top-k retrieval scores. These features characterize the difficulty of the task independent of which configuration handles it.

The meta-learner is a supervised multiclass classifier (XGBoost) trained to predict the optimal configuration from these features. "Optimal" means the configuration that clears a defined quality threshold — not the best possible answer, but the most efficient acceptable answer. This reflects the real constraint of resource-limited users: good enough at minimum compute cost. The cost function penalizes under-serving (recommending a configuration that produces a bad answer) more heavily than over-serving (recommending a slightly larger model than necessary). We use XGBoost because gradient boosted trees remain the established best practice for tabular classification at this scale (Grinsztajn et al., NeurIPS 2022; Shwartz-Ziv & Armon, Information Fusion 2022). On the question of ordinal regression (per the professor's suggestion): we use multiclass classification because the core hypothesis is that strategy can invert the expected size ordering — if a 4B model with Self-RAG beats an 8B with Naive RAG, the ordinal assumption (bigger = better) is exactly what we're trying to disprove.

In deployment, a FastAPI endpoint accepts a document and question, extracts features, and returns the recommended configuration with a confidence score. The user gets routed to the right (chunker, embedder, strategy, model) combination without manual experimentation.

Why the Pipeline Matters Beyond One Experiment

The professor's feedback suggested making the evaluation pipeline usable by others. We took this seriously. The Protocol-based architecture means a user can:

— Drop in their own documents (any CSV with title and text columns)
— Add custom embedding backends (implement three methods)
— Add custom RAG strategies (implement one method)
— Swap scoring backends (implement one method)
— Generate evaluation queries from their corpus automatically
— Get a trained meta-learner specific to their configuration space

The pipeline is the reusable contribution. The two experiments are the demonstration — showing that the same tool can answer different research questions by varying different axes. Future work — including multimodal embedding support and additional corpus types — can build on the same infrastructure without architectural changes.

Data Source

We use a sample of Wikipedia articles, providing natural variation in document length, domain, and complexity. This is a publicly available, well-understood corpus that allows comparison with published benchmarks. The sample size will be determined by experimental needs — large enough for statistical significance, small enough to run the full configuration matrix overnight on consumer hardware.

Computational Needs and Hosting

Data generation runs as an overnight batch job on a laptop with 8GB VRAM. Ollama loads quantized Qwen3 and Gemma 3 models (Q4) directly into VRAM. The pipeline is designed for sequential execution across configurations — no GPU parallelism required. The FastAPI endpoint is lightweight and will be hosted on Render.

Evaluation

We evaluate at two levels. The pipeline's output is evaluated per-configuration using Claude's faithfulness, relevance, and conciseness scores, with a manual validation sample to check for LLM-as-judge noise. The meta-learner is evaluated on held-out query-document pairs using weighted F1 and accuracy, and validated by comparing routed recommendations against the naive baseline of always using the largest model with the simplest strategy.
