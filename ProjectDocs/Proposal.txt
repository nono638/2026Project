SmallModelBigStrategy: Using RAG Strategy to Get the Most Out of Small Local Language Models Noah, Curtis, Cailinn | March 2026

The Problem
People running AI models locally — on a laptop, without a dedicated GPU — face a familiar tradeoff: small models are fast and accessible but struggle with complex questions, while larger models produce better results but are slow or impossible to run on consumer hardware. Most users default to the biggest model they can run and hope for the best.
What this misses is that how you structure the retrieval and reasoning process can matter as much as model size. Researchers have developed a range of increasingly sophisticated RAG strategies — each of which wraps different intelligence around the step where a model retrieves relevant text before answering. A smaller model using a smarter strategy may outperform a larger model using a naive one. The Self-RAG paper (Asai et al., 2024, ICLR) demonstrated exactly this on several benchmarks, but it has not been systematically studied in the sub-14B local model space with controlled experimental conditions.
Our project, SmallModelBigStrategy, tests a single hypothesis: RAG strategy sophistication can substitute for raw parameter count. We want to find out when that's true — and build a tool that learns to predict the right strategy for a given query and document.
What We're Building
The project has two components: a data generation pipeline and a trained meta-learner.
The data generation pipeline runs a large set of queries against Wikipedia articles using multiple RAG strategies across multiple locally-hosted model sizes. To isolate strategy and model size as our experimental variables, we hold everything else constant: the same embedding model, the same chunking approach, the same document corpus, and the same retrieval parameters across all runs. The only things that change between runs are the reasoning strategy and the model size.
Chunking and Embedding
We use semantic chunking (LangChain's SemanticChunker) rather than fixed-size token chunking. Fixed token chunking arbitrarily splits text mid-sentence or mid-thought; semantic chunking respects meaning boundaries. This is the approach real-world RAG deployments use, and testing against artificially degraded chunks would reduce ecological validity. Importantly, since all five strategies share the same chunks for a given document, this choice does not confound the strategy comparison — it simply makes the shared baseline more realistic.
For embeddings, we use mxbai-embed-large via Ollama. This model outperforms the commonly used nomic-embed-text on MTEB (Massive Text Embedding Benchmark) retrieval tasks while still running locally without GPU requirements. We deliberately avoid the highest-performing embedding model available: a top-tier embedder could mask strategy-level differences by making retrieval so accurate that all strategies perform similarly. mxbai-embed-large is a purposeful middle-of-the-road choice — good enough to avoid retrieval failures dominating results, not so dominant that strategy becomes irrelevant.
Our primary model family is Qwen3, which offers an unusually clean size ladder on Ollama: 0.6B, 1.7B, 4B, and 8B — all dense models, all the same architecture, all Apache 2.0 licensed and runnable locally. This gives us four well-spaced data points within a single family, eliminating architectural confounds when comparing across sizes. Llama 3.2 (1B and 3B) is included as a cross-family validation to check whether findings generalize beyond Qwen3's architecture. We use the wikimedia/wikipedia HuggingFace dataset — the same cleaned corpus used in the Adaptive-RAG benchmark — because its articles vary dramatically in length, domain, and complexity, providing natural variation in document-side features. Queries are stratified across complexity levels: simple lookup, synthesis, and multi-hop reasoning.
Each configuration produces an answer, which Claude scores as an automated judge on faithfulness, relevance, and conciseness — a methodology called LLM-as-judge that has been shown to correlate well with human judgment (Saad-Falcon et al., 2024, NAACL). These scores become our training labels.
The meta-learner is then trained to predict the optimal (strategy, model size) configuration from features of the incoming query and document. In deployment, a user pastes a document and question into a FastAPI app and gets routed to the recommended configuration — without using more compute than the query warrants.
The Five Strategies
We will test five RAG strategies, holding all other variables constant so that strategy and model size are the only things changing between runs. We explicitly exclude HyDE (Hypothetical Document Embeddings), which generates a hypothetical answer before retrieval rather than modifying the reasoning process — it's a different kind of intervention and not a fair comparison. We also exclude agentic RAG, which involves multi-step tool use and iterative reasoning loops that are implausible for models below ~7B and would introduce too many confounds.
Naive RAG — the baseline and simplest approach. The user's question is converted to an embedding, the most similar chunks from the document are retrieved, and the model generates an answer from those chunks. No special logic anywhere in the pipeline. We expect this to work fine for simple, direct questions where the answer lives in a single chunk, and to be where large models have their biggest advantage — because when the strategy provides no scaffolding, raw model capability is the only thing that matters.
Multi-Query RAG — instead of retrieving once using the original question, the model generates several different phrasings of the same question and retrieves chunks for each, then merges the results. The idea is that a question asked five ways will catch relevant chunks that one phrasing might miss. We think this could help small models specifically because it compensates for weak query understanding — a small model that doesn't fully grasp a nuanced question might still find the right content if the net is cast wider.
Corrective RAG — after retrieving chunks, a separate evaluation step scores how relevant each chunk actually is to the question. Low-scoring chunks are discarded, and if too many are discarded, a fresh retrieval is triggered. The model only sees high-confidence context when it generates. The promise is that the model is never asked to synthesize an answer from noise. We think this could help small models because their main failure mode is hallucinating when given irrelevant context — filtering that context before it reaches the model addresses the problem directly.
Adaptive RAG — before doing anything else, the query is classified by complexity: is this a simple lookup, a moderate question, or something requiring multi-step reasoning? The pipeline then routes accordingly — skipping retrieval entirely for simple questions, doing one retrieval pass for moderate ones, and doing multiple passes for complex ones. The promise is efficiency and appropriateness: not every question needs retrieval, and forcing unnecessary retrieval can actually hurt performance. We think this is the strategy most likely to benefit small models on simple queries, because it removes the retrieval step that might confuse them, letting them answer directly from their weights.
Self-RAG — the most complex strategy and the one that blurs the line between retrieval and reasoning. Rather than following a fixed pipeline, the model is prompted to decide at each step whether retrieval is needed, evaluate the relevance of what it retrieved, generate an answer, and then critique whether its own answer is well-supported. It's less a retrieval strategy and more a framework for making the model an active participant in its own pipeline rather than a passive endpoint. We implement this as a prompted approximation rather than a fine-tuned Self-RAG model, which is a practical constraint of running local models via Ollama. The promise is the highest ceiling of any strategy — but also the highest capability floor. Our most interesting hypothesis here is that Self-RAG may actually hurt very small models, because self-critique requires the model to reason about its own reasoning, which may be beyond 0.6B and 1.7B parameter models entirely. Finding that threshold is one of the things we're most curious about.
Features, Target Variable, and Cost Function
To be explicit about the machine learning problem, our feature matrix X consists of properties extracted from the query and document before any model runs:
Query length — raw token count
Query type — one-hot encoded across three categories: lookup, synthesis, and multi-hop. Query type is inherently somewhat subjective and noisy, but follows the classification scheme validated in the Adaptive-RAG literature. We classify each query using a lightweight Claude prompt as a preprocessing step — asking Claude to assign one of the three labels before the main pipeline runs. Note that Claude serves double duty here: as query classifier and as output judge. This is a acknowledged limitation; both the feature and the label carry Claude's biases, which we flag as a direction for future work.
Number of named entities in the query — a proxy for specificity and lookup complexity
Document length — token count of the source document
Document vocabulary entropy — a measure of lexical diversity, used as a proxy for how dense or technical the source material is
Mean of top-k retrieval similarity scores — how confidently the embedding model matched the query to the document
Variance of top-k retrieval similarity scores — high variance signals scattered, inconsistent context, which is a known predictor of harder inference (Zhang et al., 2025)
Our target variable Y is the minimum viable configuration — the smallest model and simplest strategy that clears a defined quality threshold on Claude's judge scores. This is a deliberate design choice. We are not trying to predict the best possible answer regardless of cost; we are trying to predict the least expensive configuration that produces an acceptable answer. This reflects the real constraint our target users face: they are on resource-constrained hardware and want good answers, not perfect ones at any compute cost.
This framing shapes our cost function. Rather than minimizing raw prediction error on quality scores, we train the meta-learner to minimize asymmetric misrouting cost: recommending a configuration that falls below the quality threshold (under-serving the user) is penalized more heavily than recommending a larger model than was strictly necessary (wasting a little compute). In plain terms, it is worse to give a bad answer than to use slightly more compute than needed. The quality threshold and penalty weights are tunable parameters that users can adjust based on their own tolerance for speed versus answer quality. (edited)[11:03 AM]Why This Is Real ML
This is a supervised multiclass classification problem with programmatically generated training data and LLM-derived labels, grounded in the Adaptive-RAG literature (Jeong et al., 2024, NAACL). For the classifier, we use XGBoost as our primary model, with Random Forest and Logistic Regression as comparison baselines. Gradient boosted trees are the established best practice for tabular classification problems at this scale — Grinsztajn et al. (NeurIPS 2022) benchmarked across 45 datasets and found they consistently outperform neural networks on tabular data, particularly under ~100K rows, and Shwartz-Ziv & Armon (Information Fusion 2022) reached the same conclusion. With ~120K rows and 9 features, a PyTorch model would add engineering complexity with likely worse generalization. The research question does not require deep learning. Training is batch initially, with online retraining as a designed extension as the app collects new labeled examples. We evaluate the meta-learner on held-out pairs using F1 and accuracy, and validate real-world usefulness by comparing routed recommendations against the naive baseline of largest-model naive RAG.
Extension: Thinking Mode as a Second Axis
Qwen3 includes a built-in reasoning toggle — models can run in standard mode or "thinking mode," where they reason step-by-step before answering. This introduces a natural extension to our hypothesis: can reasoning depth substitute for model size, independently of strategy? A 4B model in thinking mode with Self-RAG versus an 8B model in standard mode with naive RAG is a comparison nobody has cleanly studied. We note this as a stretch goal and natural future work; the pipeline supports it trivially, and it would significantly enrich the findings if time permits.

Two strategies (Naive RAG and Self-RAG), two model sizes (Qwen3 0.6B and 4B), a working Claude-scoring pipeline, a trained classifier, and a FastAPI endpoint that accepts a document and query and returns a configuration recommendation with a confidence score.
Computational Needs and Hosting
Data generation will be run as an overnight batch job on a laptop with 8GB VRAM. Ollama loads quantized Qwen3 models (Q4) directly into VRAM, making it feasible to run the full matrix of strategy and model combinations in a single session without cloud GPU resources. The FastAPI app backend is lightweight and will be hosted on Render.
