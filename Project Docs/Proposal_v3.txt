RAGBench: A Configurable RAG Evaluation Pipeline With an Application to Small Local Models
Noah Collin | March 2026 (Revised)

The Problem

There is no standard, reproducible way to answer: "which RAG configuration works best for my documents and queries?" Practitioners face a combinatorial space of chunking strategies, embedding models, retrieval approaches, and language models with no systematic tooling to explore it. Most published comparisons hold three axes constant and vary one, making results hard to generalize. Users running models locally on consumer hardware face an additional constraint: configurations must be not just good, but good enough given limited compute and time.

This project addresses both problems with three deliverables: (1) a reusable evaluation pipeline that runs the full cartesian product of RAG configurations, scoring quality and measuring latency; (2) a findings gallery with interactive visualizations from two controlled experiments on small local models; and (3) a recommendation endpoint backed by a trained meta-learner.

1. The Evaluation Pipeline

The pipeline takes four pluggable axes — chunker, embedder, strategy, and language model — and runs every combination against a document corpus with evaluation queries. Each axis is defined by a Python Protocol, so users can add implementations without modifying the framework. The result is a structured dataset of (configuration x query x document -> quality scores + latency) for analysis, visualization, and meta-learner training.

Built-in implementations span four chunkers (semantic, fixed-size, recursive, sentence), three embedders (Ollama mxbai-embed-large, HuggingFace all-MiniLM-L6-v2, Google text-embedding-005), five strategies (below), and six models: Qwen3 via Ollama (0.6B, 1.7B, 4B, 8B) as the primary family, with Gemma 3 (1B, 4B) for cross-family validation at matched parameter counts.

Evaluation Approach

The pipeline supports two coexisting evaluation modes. Intrinsic evaluation (always available) uses Claude as an LLM-as-judge, scoring each answer on faithfulness, relevance, and conciseness — no gold-standard answers required. Extrinsic evaluation (when gold data exists) compares answers against reference answers using exact match and token-level F1, providing an independent correctness signal and enabling scorer validation.

Per the professor's feedback on LLM-as-judge noise: we validate the scorer empirically by correlating Claude's ratings with gold-answer correctness on HotpotQA, rather than relying on assumption.

Every run also measures wall-clock latency for both strategy execution and scoring, enabling tradeoff analysis: "is the 60-second answer meaningfully better than the 5-second one?"

Data Sources

Primary corpus: HotpotQA (Yang et al., 2018), a multi-hop Wikipedia QA dataset with 113K examples. Each contains 10 passages (2 supporting + 8 distractors), a gold answer, question type labels (bridge/comparison), and difficulty levels (easy/medium/hard). Multi-hop questions are where strategies should diverge most — bridge questions require chained retrieval, which is exactly where Corrective RAG and Adaptive RAG should outperform Naive RAG.

Secondary corpus: a 10,000-article Wikipedia sample with template-generated queries (no gold answers, intrinsic evaluation only), serving as a generalization check.

Query generation is pluggable: we implement RAGAS synthetic generation, template-based generation with NER, BEIR benchmark loading, and human-curated query sets, with round-trip consistency and cross-encoder filters for quality validation.

The Five Strategies

Naive RAG — retrieve chunks, generate answer. Baseline where raw model capability matters most.

Multi-Query RAG — generates multiple query phrasings, retrieves for each, merges results. Compensates for weak query understanding in small models.

Corrective RAG (Shi et al., 2024) — evaluates chunk relevance and discards low-confidence context before generation. Addresses small models' tendency to hallucinate from irrelevant context.

Adaptive RAG (Jeong et al., 2024) — classifies query complexity and routes: skip retrieval for simple, single-pass for moderate, multi-pass for complex.

Self-RAG (Asai et al., 2024) — the model decides when to retrieve, evaluates what it retrieved, generates, and self-critiques. Highest ceiling but requires self-reasoning capability that very small models may lack entirely.

Experiment 1: Strategy x Model Size

Research question: can RAG strategy sophistication substitute for raw parameter count?

We hold chunker and embedder constant and run the 5 x 4 matrix of strategies x Qwen3 sizes, plus 5 x 2 for Gemma 3 — 30 configurations per query. The hypothesis is that strategies like Self-RAG and Corrective RAG disproportionately help small models by compensating for their weaknesses, while providing diminishing returns for larger models that handle these challenges natively.

Experiment 2: Chunking Strategy x Model Size

Research question: does the optimal chunking strategy depend on model size?

Existing chunking comparisons (Wang et al., 2024; Chen et al., 2023) use only large models (GPT-3.5/4). No published work studies how chunking interacts with model size for small (<14B) models. Liu et al. (2023) showed that small models degrade more with irrelevant context, suggesting chunking precision matters more for them.

We hold strategy (Naive RAG) and embedder constant, running 3 chunkers x 4 Qwen3 sizes — 12 configurations per query. The hypothesis is that smaller models benefit disproportionately from precision-maximizing chunking (semantic > recursive > fixed), with practical implications for users running small local models.

Meta-Learner

From each (query, document, retriever) triple, we extract features before any model runs: query length, query type, named entity count, document length, vocabulary entropy, and top-k retrieval score statistics. An XGBoost classifier predicts the optimal configuration — the most efficient one that clears a quality threshold, not the absolute best. We use multiclass classification rather than ordinal regression (per the professor's suggestion) because the core hypothesis is that strategy can invert the expected size ordering: a 4B with Self-RAG may beat an 8B with Naive RAG.

2. The Findings Gallery

A web-based presentation of experiment insights with interactive visualizations, designed for practitioners curious about RAG tradeoffs. The gallery surfaces cases where small models with the right strategy outperform larger ones, time-vs-quality tradeoff curves showing where additional compute stops buying improvement, the impact of individual decisions in experimental isolation, and difficulty-dependent behavior across strategies.

3. The Recommendation Endpoint and Reusable Package

A FastAPI endpoint accepts a document and question, extracts features, and returns the recommended configuration with a confidence score, hosted on Render. The pipeline is also designed as a reusable tool: users can drop in their own documents, add custom embedders or strategies, run experiments on built-in datasets (HotpotQA) to calibrate before testing their own data, and define what "best" means for their use case — optimizing for quality, latency, cost, or any combination.

Computational Needs

Data generation runs as an overnight batch job on a laptop with 8GB VRAM. Ollama loads quantized Qwen3 and Gemma 3 models (Q4) directly into VRAM, with sequential execution across configurations.

Evaluation

Per-configuration quality: Claude's scores validated against gold-answer correctness on HotpotQA. Per-configuration latency: wall-clock timing enabling time-quality tradeoff analysis. Meta-learner accuracy: weighted F1 on held-out pairs, validated against the naive baseline of always using the largest model with the simplest strategy.
