How This Experiment Works

We start with HotpotQA, a dataset of real questions where the correct answers are already known. Each question comes with source documents and a verified "gold" answer. We feed the question and documents into a RAG pipeline (NaiveRAG + Qwen3 4B) to generate an answer, then measure that answer two ways:

  1. Automated metrics compare the RAG answer to the gold answer using BERTScore (semantic similarity) and F1 (word overlap) — these are objective, but can't capture everything.
  2. LLM judges (6 different cloud models) read the question, context, and RAG answer, then rate quality on faithfulness, relevance, and conciseness — without ever seeing the gold answer.

The question this experiment answers: which judges' ratings actually track the objective metrics? If a judge says an answer is good, is it really good?

How We Measured "Correctness"

The charts above keep referring to "BERTScore" and "F1" — these are the two automated metrics we use to measure how close a RAG answer is to the known-correct gold answer. They measure different things:

  • BERTScore (0–1): Uses a neural language model to measure semantic similarity. "Steve McQueen" and 'Terence Steven "Steve" McQueen' score high because they mean the same thing. This is our primary metric.
  • F1 (0–1): Counts shared words between the RAG answer and the gold answer. Strict and literal — penalizes correct paraphrases. Used as a secondary check to confirm BERTScore isn't being fooled.

Why two metrics? If both agree, we're confident. If they diverge, the model is paraphrasing (high BERTScore, low F1) — which is fine, but worth knowing.

Lessons Learned

What We Got Wrong in Experiment 0 (v1)

Experiment 0 was our first end-to-end pipeline run. It answered the question it was designed to answer — which scorer to trust — but post-analysis revealed five methodological oversights that we are addressing in Experiment 0 v2.

1. We didn't capture what the LLM saw

The pipeline generated answers but didn't record which chunks were retrieved, what context was assembled, or what the LLM actually received as input. When the Church of St. Anne question failed (example 5), we could see the answer was wrong but couldn't determine why — was it a retrieval miss, a chunking problem, or a generation error? Without pipeline observability, failure analysis is guesswork.

Fix: Pipeline diagnostics now capture retrieved chunks, filtered chunks, context sent to the LLM, and automatically attribute failures to the responsible pipeline stage (chunker, retrieval, filtering, or generation).

2. The scorer judged against information the LLM never received

When scoring each answer, we passed the full source document as context to the LLM judge. But the answering model only saw the retrieved chunks — a small subset of the document. This means the faithfulness score measured whether the answer was consistent with the entire document, not with what the model actually had access to. A hallucinated detail that happened to appear elsewhere in the document would score as “faithful.”

Fix: v2 passes the actual context sent to the LLM (context_sent_to_llm) to the scorer, so faithfulness is evaluated against what the model truly saw.

3. No reranker in the pipeline

The v1 pipeline used raw hybrid retrieval (dense + BM25 with RRF fusion) without a cross-encoder reranker. This is a weaker pipeline than what Experiments 1 and 2 will use, which means we validated our scorers on a different pipeline configuration than the one they'll actually score. Reranking improves retrieval precision and changes the distribution of answer quality — scorer validation should reflect the real pipeline.

Fix: v2 adds BGE Reranker v2 M3 (568M parameters) as the default reranker. Retrieve 10 candidates, rerank down to 3.

4. Ceiling effect — too many easy questions

With 50 HotpotQA questions sampled proportionally across difficulties, 74% of answers were correct (exact match). Most judges rated most answers 5/5 — Flash-Lite gave a perfect 5.0 on 78% of examples, Opus on 78%. With only ~13 wrong answers, there wasn't enough signal to meaningfully compare how well judges discriminate between good and bad answers.

Fix: v2 increases the sample to 150 questions and filters to medium and hard difficulty only (dropping easy questions entirely). This produces more wrong and partial answers, giving us real statistical power to compare scorer discrimination.

5. No composite answer quality metric

We had three independent signals for answer correctness — BERTScore (semantic similarity to the gold answer), word-overlap F1, and LLM judge scores — but no way to ask: “do all three agree this answer is good?” A judge that gives 5/5 to an answer with low BERTScore and low F1 has a blind spot. Example 31 illustrates this perfectly: the RAG answer was “Not specified in the context” (a polite refusal), Flash and Opus both gave it 5/5 for faithfulness (it was faithful to the empty context), but the gold answer was “El Alma Argentina” — a complete miss.

Fix: v2 adds an answer_quality column that requires agreement across all three metrics: BERTScore ≥ 0.90, word-overlap F1 ≥ 0.50, and Sonnet quality ≥ 4.0. An answer is only “good” if the gold metrics and the best judge all agree. This triangulation exposes the blind spots that any single metric misses.

Experiment 0 v2 reruns this scorer validation with all five fixes in place. The v1 results above are preserved as-is — they are the baseline that motivated these improvements.
